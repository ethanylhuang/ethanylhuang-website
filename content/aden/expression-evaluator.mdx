## Objective

I thought the Expression Evaluator project was interesting because it forced me to algorithmically represent the process of calculating an expression. This is something we do almost as second nature, but applying it to computers meant explicitly handling operation order, implicit multiplication, and invalid expressions.

## Project Phases

The project started with a stack implementation for basic operations. I used one stack for numbers and one stack for operators, so the evaluator could delay execution until operator precedence and parentheses made it valid to calculate. This was the first time I had to turn something that feels automatic in math into a strict algorithm.

After that, I added static error checking, especially for parentheses. The program checked for mismatched parentheses, empty parentheses, invalid operator placement, invalid data tokens, and divide-by-zero at runtime. This mattered because the evaluator depended on clean tokens before the stack loop could safely run.

The later phases were more complicated because they dealt with cases that look natural to people but are not explicit in code. I added support for implicit multiplication, implicit negatives, and negative numbers in general. A minus sign could mean subtraction or a negative number depending on where it appeared, and expressions like `2(3+4)` had to be converted into something the evaluator could actually process.

The main loop came down to deciding when the top operator should execute. This small rule carried most of the order-of-operations behavior.

```java
if (operStack.empty() || operStack.peek().equals("(") || curr.equals("(")) {
  return false;
}
if (operStack.peek().matches("[\\+\\-]") && curr.matches("[\\*\\/]")) {
  return false;
}
return true;
```

## Debugging The Evaluation Loop

This was probably the most challenging project for me in the class so far, simply because of the layers of complexity. I had a full version passing the `basic` and `basic_err` tests, but advanced expressions with implicit multiplication started breaking older behavior.

After stepping through the debugger, I found a fundamental flaw in my core evaluation loop. I tried to patch it with small fixes, but the logic got more convoluted because the original structure was less than optimal. Eventually, I cut my losses and rewrote the basic evaluation loop from scratch, using what I had learned to make it simpler and cleaner.

## Learnings

The biggest learning was that clean core logic matters more than passing the first tests quickly. A shaky evaluation loop can survive simple cases, but it becomes painful once edge cases start stacking on top of it.

I was proud of the final product because I think it was some of the cleanest code I had written yet. It also made me wonder how similar computer math engines work under the hood, especially in systems where the order of operations has to stay deterministic.
