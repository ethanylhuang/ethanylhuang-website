## Objective

During this project, I learned how Huffman compression works by using variable length character representations, with the most frequent characters getting the shortest codes. The goal was to build that structure from the ground up, then use it to compress and analyze real text files.

## Compression Tree And Binary IO

The implementation centered on generating character weights, building a frequency weighted tree, and using binary IO to write and read compressed data. This takes advantage of the fact that characters like spaces or common letters show up often, so giving them shorter representations saves space.

I liked the pacing of the project because it let me work through the implementation section by section after being taught the concepts in class. Going from weights, to the tree, to actual compression made the algorithm feel understandable instead of appearing all at once.

## Data Analysis

The compression ratio analysis made the tradeoffs more concrete. A weights file generated from a specific text should be optimal for that text, but a similar text can still benefit from the same structure.

The Cat in the Hat weights worked better for most files except War and Peace. This made sense because Green Eggs and Ham and The Cat in the Hat are both poems, meaning newlines and punctuation were weighted more heavily and given shorter binary representations. Harry Potter had roughly equal compression ratios because it is a novel, so it did not benefit as much from efficient compression for newlines and punctuation.

## Learnings

I found it interesting that such a fundamental and common compression algorithm is relatively simple at its core. It also made the memory tradeoff more visible, since the Huffman tree has to be stored in order to decode the file.

I still wonder about the efficiency limits of Huffman compression and how similar algorithms are used for image compression.
